# Code Reader Skill — 代码阅读与解释

## 任务说明

你是一名资深软件工程师，负责阅读并详细解释一个代码库。你需要：

1. 扫描目标目录中的所有源代码文件
2. 为**每个源代码文件**生成详细说明文档（存入 `./doc_output/`）
3. 为**每个目录**生成简略概述文档
4. 最后生成一份整体架构与流程总结文档
5. 默认输出语言：简体中文

目标目录：`$ARGUMENTS` （若为空则为当前工作目录 `.`）

---

## 进度追踪机制

本技能使用进度文件 `doc_output/PROGRESS.md` 来跟踪所有任务的状态。状态标记如下：

- ⏳ **待处理**：尚未开始分析的文件/目录
- 🔄 **进行中**：正在分析的文件/目录
- ✅ **已完成**：已生成文档的文件/目录
- ⏭️ **跳过**：被排除或无需处理的文件/目录

**工作流程**：
1. 首次运行时，扫描目标目录并创建进度文件，同时检查 `doc_output/` 中已存在的文档，标记为 ✅ 已完成
2. 每次处理前，从进度文件中找出第一个 ⏳ 待处理项，将其改为 🔄 进行中
3. 完成分析后，将该项标记为 ✅ 已完成
4. 重复步骤 2-3 直到所有项都完成

---

## 第一步：扫描代码库结构并初始化进度

### 1.1 扫描源代码文件

使用 Glob 和 Bash 扫描目标目录，列出：
- 所有源代码文件（`.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.go`, `.rs`, `.java`, `.c`, `.cpp`, `.cs`, `.rb`, `.php`, `.swift`, `.kt`, `.sh`, `.yaml`, `.yml`, `.json`, `.toml`等）
- 目录层级结构

排除以下目录和文件：
- `node_modules/`, `dist/`, `build/`, `.git/`, `__pycache__/`, `.venv/`, `venv/`, `vendor/`, `target/`, `bin/`, `obj/`
- `*.min.js`, `*.min.css`, `*.map`, `*.lock`, `package-lock.json`, `yarn.lock`
- `doc_output/`（避免递归处理自己的输出）

```bash
TARGET="${ARGUMENTS:-\.}"
find "$TARGET" \
  -not \( -path "*/node_modules/*" -o -path "*/.git/*" -o -path "*/dist/*" -o -path "*/build/*" -o -path "*/__pycache__/*" -o -path "*/.venv/*" -o -path "*/venv/*" -o -path "*/vendor/*" -o -path "*/target/*" -o -path "*/doc_output/*" \) \
  -type f \( \
    -name "*.ts" -o -name "*.tsx" -o -name "*.js" -o -name "*.jsx" \
    -o -name "*.py" -o -name "*.go" -o -name "*.rs" \
    -o -name "*.java" -o -name "*.c" -o -name "*.cpp" -o -name "*.cs" \
    -o -name "*.rb" -o -name "*.php" -o -name "*.swift" -o -name "*.kt" \
    -o -name "*.sh" -o -name "*.bash" \
    -o -name "*.yaml" -o -name "*.yml" -o -name "*.toml" \
    -o -name "*.json" -not -name "package-lock.json" -not -name "yarn.lock" \
    -o -name "*.md" -o -name "Makefile" -o -name "Dockerfile" \
  \) \
  | sort
```

### 1.2 扫描目录结构

```bash
TARGET="${ARGUMENTS:-\.}"
find "$TARGET" \
  -not \( -path "*/node_modules/*" -o -path "*/.git/*" -o -path "*/dist/*" -o -path "*/build/*" -o -path "*/__pycache__/*" -o -path "*/doc_output/*" \) \
  -type d | sort
```

### 1.3 初始化进度文件

创建 `doc_output/PROGRESS.md` 文件，记录所有待处理的文件和目录。

首先检查输出目录中已有的文档：

```bash
# 获取已完成的文件（.md 文件存在且对应源文件也存在）
find ./doc_output -name "*.md" -not -name "PROGRESS.md" -not -name "SUMMARY.md" -not -name "README.md" 2>/dev/null | sed 's|^./doc_output/||' | sed 's|\.md$||'

# 获取已完成的目录（README.md 存在）
find ./doc_output -name "README.md" -not -path "./doc_output/README.md" 2>/dev/null | sed 's|^./doc_output/||' | sed 's|/README\.md$||' | sed 's|README\.md$|.|'
```

创建进度文件模板：

```markdown

> 生成时间：[当前时间]

---

|------|----------|------|

| [状态] | [目录2] | |

| 状态 | 文件路径 | 备注 |
|------|----------|------|

---

## 📝 汇总

- ⏳ 待处理：[N] 个
- 🔄 进行中：[N] 个
- ✅ 已完成：[N] 个
- ⏭️ 跳过：[N] 个
- **总计**：[N] 个

---
```

- 如果 `doc_output/[目录]/README.md` 已存在 → 标记为 ✅ 已完成
- 否则 → 标记为 ⏳ 待处理

## 执行要求

- **进度优先**：每次处理前必须先读取 `PROGRESS.md`，找出第一个 ⏳ 待处理项
- **状态更新**：开始分析前将状态改为 🔄 进行中，完成后改为 ✅ 已完成
- **逐文件处理**：每次 Read 一个文件，立刻 Write 对应文档，不要批量处理
- **真实内容**：所有说明必须基于实际读取的代码内容，不能凭空捏造
- **中文输出**：所有文档均用中文撰写（代码片段和专有名词保持原文）
- **深度分析**：每个文件的说明要足够详细，让读者无需查看源代码就能理解该文件的作用
- **保持链接**：目录 README 中的文件链接要正确指向对应的 .md 文档
- **进度反馈**：每完成一个文件，输出简短进度信息（例如：`✅ 已完成：src/utils/helper.ts`）并更新 PROGRESS.md

## 完成标志

所有文件处理完毕后，输出：

```
===== 代码阅读完成 =====
共分析文件：[N] 个
共分析目录：[M] 个

整体总结：./doc_output/SUMMARY.md
进度追踪：./doc_output/PROGRESS.md
=========================
```

最终 `doc_output/PROGRESS.md` 应该显示所有项都是 ✅ 已完成状态。
